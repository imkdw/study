# 핵심에서 일반으로

- 택배사의 배송 경로를 최적화하는 혁신적인 알고리즘을 개발해서 경쟁사보다 낮은 배송료를 청구하는 업체가 있다
- 경쟁사에서 `외판원 문제`를 해결하고 더 낮은 비용으로 서비스를 제공하기 시작한다
- 이런 경우는 기존 핵심이 도메인이 일반 도메인으로 바뀌게된다. 경쟁사의 솔루션을 따라잡기 위해서는 막대한 비용투자가 필요하다

<br>

# 일반에서 핵심으로

- 재고 관리를 위해서 상용 솔루션을 사용하고 있었지만 계속해서 잘못된 지표가 보이게된다
- 결국 경영진은 사내 시스템 설계 및 구축에 투자하기로 전략적 결정을 내리게된다
- 이러한 사내 솔루션은 제품의 복잡성을 고려해서 고객의 수요를 더 잘 예측하게된다
- 기존 일반 도메인에서 사용하던 솔루션이 핵심 도메인으로 바뀌게된다
- 대표적으로 아마존이 있는데 기존 시스템을 올릴 인프라가 필요해서 직접 구축하고 이를 수익성 비즈니스인 AWS로 전환했다

<br>

# 지원에서 일반으로

- 마케팅 부서에서 협력하는 공급업체와 계약을 관리하기 위한 시스템을 구현한다
- 기존에는 데이터 입력을 위한 단순 CRUD 인터페이스만 존재했다. 즉 전형적인 지원 하위 도메인이다
- 사내 솔루션을 구현하기 시작한지 몇 년 후 계약 관리 솔루션이 오픈소스로 탄생했는데 이를 도입하기로 한다
- 이러면 문서 관리 솔루션의 하위 도메인이 지원 도메인에서 일반 도메인으로 변경된다

<br>

# 지원에서 핵심으로

- 회사에서 비용을 줄이거나 추가 수익을 창출하는 방식으로 지원 로직을 최적화하는 방법을 찾는 경우가 이에 해당된다
- 일반적인 전조 증상은 항뤼 도메인의 비즈니스 로직이 점점 더 복잡해지는 것이다
- 하지만 회사의 이익에 영향을 주지 않는데 더 복잡해지는 이유는 우발적 비즈니스 복잡성 때문이다
- 만약 회사의 수익성이 개선된다면 이는 지원 하위 도메인이 핵심 하위 도메인으로 전환될 징조다

<br>

# 핵심에서 지원으로

- 하위 도메인의 복잡성이 정당화되지 않을 때 발생할 수 있다
- 복잡성에 비해서 수익성이 없는 경우인데, 다른 하위 도메인의 개발을 지원하는 데 필요한 최소한의 로직만 남기고 복잡성을 줄이기로 결정하는 경우다

<br>

# 일반에서 지원으로

- 이전 사례에서 오픈소스 솔루션과 통합 과정에서 복잡성 대비 얻는 이점을 정당화 될 수 없다고 결정을 내렸다고 가정한다
- 이런 경우는 다시 일반 하위 도메인이 지원 하위 도메인으로 전환된다
